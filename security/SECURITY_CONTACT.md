# Security Contact and Vulnerability Disclosure Policy

## Reporting

Security issues should be reported privately when possible.

- Security contact email: `security@example.com`

If a private reporting channel is not yet configured, use a private email channel rather than a public GitHub issue for suspected credential exposure, authentication bypass, or sensitive data exposure.

## Requested report contents

- Affected repository and branch
- Clear description of the issue
- Steps to reproduce, if available
- Potential impact
- Any proof-of-concept artifacts that are necessary to validate the issue

## Response expectations

- Reports will be acknowledged as soon as practical.
- Critical reports should be triaged immediately upon review.
- Valid reports will be investigated, remediated, and disclosed in a manner appropriate to the severity and exposure.

## Safe disclosure expectations

- Do not publicly disclose a vulnerability until the maintainer has had a reasonable opportunity to investigate and remediate it.
- Do not access, modify, or exfiltrate data beyond what is necessary to demonstrate the issue.
- Do not attempt denial-of-service, destructive testing, or social engineering.

## Public disclosure

- After remediation, material issues may be disclosed through a GitHub advisory, release note, repository notice, or equivalent public communication.
