Third-Party Services and Subprocessors

Purpose

This document summarizes the main third-party services referenced by the reviewed repositories and the control expectations that apply to them.

Categories observed in reviewed repositories

- Source control and CI/CD: GitHub, GitHub Actions, Dependabot
- Secret management: GCP Secret Manager, Azure Key Vault, 1Password
- Cloud hosting and infrastructure: Google Cloud Run, Azure Databricks, documented AWS ECS patterns
- External APIs and platforms: Microsoft Graph, SharePoint, Snowflake

Control expectations

- Third-party services must be accessed using individual user identities, managed identities, service accounts, or provider-approved credentials.
- Production secrets for third-party services must be stored only in managed secret stores or equivalent approved secure tooling.
- Access to third-party administrative consoles must be limited to the maintainer and explicitly authorized collaborators.
- Logs and diagnostics sent to third-party platforms must not include secrets, access tokens, authorization headers, or raw sensitive financial payloads.

Plaid-specific expectation

- If Plaid is added to the environment, Plaid credentials, access tokens, and retrieved financial data must remain outside public source control and must be processed only in controlled runtime environments.

Review cadence

- This document must be reviewed at least annually and after any material change to hosting, CI, secret storage, or external integrations.
